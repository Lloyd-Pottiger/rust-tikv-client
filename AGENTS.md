# Repository Guidelines

## Project Structure

- `src/`: `tikv-client` Rust library sources.
- `src/generated/`: protobuf/gRPC bindings generated by `proto-build` (do not edit by hand).
- `proto/`: upstream TiKV/PD `.proto` files; `proto/include/` contains shared includes.
- `proto-build/`: code generator crate that regenerates `src/generated/` via `tonic-build`.
- `tests/`: Rust test suites.
  - `tests/integration_tests.rs`: requires a running PD + TiKV cluster.
  - `tests/failpoint_tests.rs`: requires `--features fail/failpoints`.
- `examples/`: runnable client examples.
- `config/`: `nextest` config and TiUP playground configs (`tikv.toml`, `pd.toml`).
- `client-go/`: Go TiKV client (separate Go module with its own `go.mod`).

## Commands

- Generate protobufs: `make generate` (requires `protoc`).
- CI-like checks: `make check` (runs generate, `cargo check`, `cargo fmt --check`, `cargo clippy`).
- Unit tests: `make unit-test` (uses `cargo nextest`; install with `cargo install cargo-nextest --locked`).
- Integration tests: `make integration-test` (requires a running cluster; uses `$PD_ADDRS`).
- Start local cluster: `make tiup` (runs a TiUP playground in the background).
- Docs: `make doc`
- Clean: `make clean`

## Workflow Principles

- Don’t edit `src/generated/` directly; update `.proto` files or `proto-build/`, then run `make generate`.
- Keep changes minimal and consistent with existing patterns; avoid drive-by refactors.
- Treat warnings as errors (`Makefile` sets `RUSTFLAGS=-Dwarnings`); keep code warning-free.
- Prefer targeted tests over full-suite runs; avoid running multiple `cargo`/`nextest` commands concurrently (Cargo cache lock contention).
- Be careful with async code: avoid blocking calls on Tokio threads and never hold locks across `.await`.

## Engineering Rules

This project is a public client library. Optimize for correctness, API stability, and maintainability.

### Core Principles: Simplicity & Readability

- Prefer boring, explicit code over clever abstractions.
- Keep functions small and single-purpose; name things so they need minimal explanation.
- Follow existing patterns in this repo unless there is a clear reason to change them.

### Library/API Quality

- Avoid breaking changes to public APIs unless explicitly required; prefer additive changes and backward-compatible defaults.
- Public APIs should not `panic!`; return `Result` with meaningful errors (no `unwrap/expect` in library code).
- Document public items and non-obvious behavior; update `README.md`/`getting-started.md`/`examples/` when user-facing behavior changes.

### Async/Concurrency Correctness

- Do not block the Tokio runtime (no long CPU work or blocking IO on async tasks).
- Avoid holding locks across `.await`; be deliberate about cancellation and retries/backoff behavior.
- Keep request handling and retry logic deterministic and well-tested.

### Dependencies, Safety, and Performance

- Be conservative adding dependencies (size, build time, MSRV/toolchain impact); reuse existing crates when possible.
- Avoid `unsafe`; if unavoidable, isolate it and document the invariant it relies on.
- Minimize unnecessary allocations/clones in hot paths (requests, retries, codecs); benchmark or add targeted tests when performance-sensitive.

### Testing & Tooling

- Add/adjust tests for behavior changes; prefer targeted unit tests and focused integration tests.
- Keep `make check` and formatting clean before finishing work.

## Integration Test Notes

- `PD_ADDRS` is a comma-separated list of PD endpoints (default: `127.0.0.1:2379`).
- Multi-region test setup is enabled when `MULTI_REGION` is present in the environment (the `Makefile` sets it by default). To run with a single region, unset it, e.g. `env -u MULTI_REGION make integration-test`.

## API / Docs Expectations

- This is a library crate: keep the public API small and well-documented; avoid breaking changes unless explicitly required.
- Update `README.md`, docs, and examples when changing user-facing behavior or configuration.

## Go Client (`client-go/`)

- Format: `gofmt -w .` (or `go fmt ./...`).
- Test: `go test ./...`
